Adapter Pattern

Converts the interface of a class into another interface clients expect — letting incompatible interfaces work together.

Theory

The problem, in plain terms

Your app's payment module expects every provider to implement a PaymentProcessor interface with a pay(amount) method. You need to integrate a third-party SDK that exposes makeTransaction(cents, currencyCode) instead — a completely different shape. You can't change the SDK, and you don't want if (usingThirdPartySDK) branches scattered through your payment module.

An Adapter sits between the two: it implements the interface your code expects, and internally translates each call into whatever the incompatible library actually needs. Your payment module never knows it's talking to a translator.

How it's built

Two common shapes: object adapter (the adapter holds a reference to the wrapped "adaptee" and delegates to it, translating arguments as needed) and class adapter (the adapter inherits from both the target interface and the adaptee — only possible with multiple inheritance, so rare in Java, more common in C++). Object adapter is the portable, commonly used form.

✓ Good practice

Keep the adapter thin — its only job is translating shape (method names, argument formats, units). If it starts adding business logic, that logic belongs somewhere else.

✕ Common mistake

Confusing this with Facade. Adapter matches a specific, pre-existing interface your code expects. Facade just simplifies access to something complex — it isn't matching anything that already exists.

Where it bites you

Every adapted dependency is a translation layer to maintain — if the adaptee's API changes, the adapter changes with it. And an adapter can only fix a mismatch in interface shape; it can't fix a fundamental mismatch in behavior (a synchronous adaptee behind an interface your code assumes is async, for instance).

Diagram

When to Use

✓ Reach for it when
  • You need to integrate a third-party/legacy interface that doesn't match what your code expects, and can't modify its source.
  • You want to isolate third-party specifics in one translation layer instead of scattering them through the codebase.
✕ Don't reach for it when
  • You control both interfaces — just change one to match the other.
  • The mismatch is behavioral, not just structural — an adapter can't fix that.

Real-World Examples

  • Java's Arrays.asList() — adapts a plain array to the List interface.
  • InputStreamReader / OutputStreamWriter — adapts byte streams to character-oriented Reader/Writer interfaces.
  • Physical power adapters — the canonical non-software example: converts plug shape/voltage without changing the device or the wall.
  • Third-party SDK wrappers (payment, SMS, analytics) — nearly every production codebase wraps vendor SDKs so the vendor can be swapped later.
  • Cross-browser JS polyfills/shims — adapting inconsistent browser APIs to one consistent interface.

Interview Questions

Click a question to reveal the answer.

What's the difference between an object adapter and a class adapter?

An object adapter holds a reference to the adaptee and delegates to it (composition) — works in any OOP language, and is the more common form. A class adapter inherits from both the target interface and the adaptee (multiple inheritance) — only viable in languages that support it, like C++; not possible in Java.

How is Adapter different from Facade?

Adapter makes an existing interface match a specific interface your code already expects — one-to-one compatibility. Facade simplifies access to a complex subsystem behind one new, simpler interface — it isn't matching anything pre-existing, it's inventing an easier one.

How is Adapter different from Decorator, given both "wrap" another object?

Adapter changes the interface — the wrapped object couldn't be used directly because its methods don't match what's expected. Decorator keeps the same interface as the object it wraps, and adds new behavior on top — the wrapped and wrapping object are interchangeable from the caller's perspective, which is never true for Adapter's adaptee.

Can an Adapter wrap multiple different adaptees behind the same target interface?

Yes — common in practice. You might have StripeAdapter, PaypalAdapter, and RazorpayAdapter all implementing the same PaymentProcessor interface, each translating to a different vendor SDK. Calling code depends only on PaymentProcessor and never knows which vendor it's actually using.
