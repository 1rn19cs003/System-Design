System Design Glossary

Sixty-plus terms from HLD, LLD, and design patterns, defined in one or two sentences each. Meant for scanning, not reading top to bottom — use the A–Z bar or your browser's find (Ctrl/Cmd+F) to jump straight to a term.

ABCDE FGHIJ KLMNO PQRST UVWXYZ

A

ACID
Atomicity, Consistency, Isolation, Durability — the four guarantees a traditional relational transaction makes, so a multi-step operation either fully completes or fully rolls back.
API Gateway
A single entry point in front of a set of services that handles routing, auth, rate limiting, and request shaping, so clients don't talk to each backend service directly.
At-least-once delivery
A messaging guarantee that a message will be delivered one or more times, never zero — safe against loss, but requires the consumer to handle duplicates (idempotency).
At-most-once delivery
A messaging guarantee that a message is delivered zero or one times, never more — safe against duplicates, but can silently lose messages.
Availability
The fraction of time a system is usable, often expressed as a string of nines (99.9%, 99.99%). See Core Principles.

B

Back-of-envelope estimation
Quick, order-of-magnitude math (traffic → QPS → storage → servers) used early in a system design interview to size the problem before designing. See the Numbers Every Engineer Should Know page.
Backpressure
A mechanism where a slow downstream consumer signals an upstream producer to slow down, preventing the consumer from being overwhelmed.
Bloom filter
A space-efficient probabilistic structure that can say "definitely not present" or "maybe present" for set membership, used to avoid expensive lookups for keys that don't exist.
Bounded context
A domain-driven design boundary within which a particular model and its terms have one consistent meaning — often maps directly onto a microservice boundary.

C

Cache-aside
A caching pattern where the application checks the cache first, and on a miss reads from the database and populates the cache itself. See Caching.
CAP theorem
During a network partition, a distributed system must choose between Consistency and Availability — partition tolerance itself isn't optional on a real network. See Core Principles.
CDN (Content Delivery Network)
A geographically distributed network of caching servers that serve static (and some dynamic) content from a location physically close to the requester, cutting latency and origin load.
Circuit breaker
A pattern that stops calling a failing downstream dependency for a cooldown period after repeated failures, preventing cascading failure and giving the dependency time to recover.
Consensus
The problem of getting a set of distributed nodes to agree on a single value or decision even in the presence of failures — solved by algorithms like Paxos and Raft.
Consistent hashing
A hashing scheme that minimizes data movement when nodes are added or removed from a distributed cache or database, by mapping both keys and nodes onto the same ring. See Databases.
Cursor-based pagination
Pagination that uses an opaque pointer to the last-seen item instead of an offset/page number, avoiding the performance and consistency problems of large offsets.

D

Denormalization
Deliberately duplicating data across tables/documents to avoid expensive joins at read time, trading storage and write complexity for read speed.
Distributed lock
A lock coordinated across multiple machines (often via a service like ZooKeeper, etcd, or Redis) so only one node performs a critical section of work at a time.
DNS (Domain Name System)
The distributed, eventually-consistent system that translates human-readable domain names into IP addresses. See HLD Fundamentals.
Durability
The guarantee that once a write is acknowledged, it survives crashes, power loss, or restarts — usually achieved by writing to disk and/or replicating before acknowledging.

E

Eventual consistency
A model where replicas may briefly disagree after a write but converge given enough time and no new writes — trades immediate correctness for availability and speed.
Exponential backoff
A retry strategy where the wait time between retries grows exponentially (often with random jitter added) to avoid overwhelming a recovering service.

F

Fan-out
Delivering one piece of data to many recipients — e.g., pushing a new tweet to every follower's feed. See the Twitter / X case study.
Fault tolerance
A system's ability to keep functioning correctly even when one or more of its components fail.
Federation
Splitting a single large database into multiple databases by function (e.g., users DB, orders DB) rather than by row — a simpler alternative to full sharding.

G

Gossip protocol
A peer-to-peer communication style where nodes periodically exchange state with a few random peers, spreading information across a cluster without central coordination.

H

Heartbeat
A periodic signal a node sends to indicate it's still alive; missed heartbeats are how load balancers and cluster managers detect failures.
Horizontal scaling
Adding more machines to handle more load, as opposed to making one machine bigger. See Core Principles.
Hot partition / hotspot
A single shard or key that receives disproportionately more traffic than the others, becoming a bottleneck even though the system is "scaled."

I

Idempotency
A property where performing the same operation multiple times has the same effect as performing it once — critical for safely retrying requests.
Indexing
A data structure (commonly a B-tree or hash index) that lets a database find rows without scanning the whole table, at the cost of extra storage and slower writes.
Isolation level
How much one database transaction is shielded from the effects of concurrent transactions (e.g., Read Committed, Repeatable Read, Serializable) — stronger isolation costs concurrency.

J

JWT (JSON Web Token)
A signed, self-contained token format commonly used to carry authentication/authorization claims between services without a shared session store.

K

Key-value store
A database that maps unique keys to values with no required schema on the value — optimized for very fast lookups by key (e.g., Redis, DynamoDB).

L

Latency
The time it takes for a single request to complete, from sent to received. See the Latency Numbers reference.
Leader election
The process by which a distributed cluster picks one node to act as the coordinator/primary, so writes or decisions aren't made by multiple nodes at once.
Load balancer
A component that distributes incoming requests across multiple backend servers, improving both throughput and fault tolerance. See Load Balancing.
Long polling
A technique where the client requests data and the server holds the connection open until new data is available (or a timeout), simulating push over plain HTTP.
LRU cache
"Least Recently Used" — a cache eviction policy that discards the item that hasn't been accessed in the longest time when the cache is full. See Caching.

M

Message queue
An intermediary that buffers messages between producers and consumers, decoupling them in time and allowing async, resilient processing. See Message Queues.
Microservices
An architecture style that splits an application into small, independently deployable services, each owning its own data. See Microservices.
Monolith
An application built and deployed as a single unit — simpler to develop and deploy early on, harder to scale specific parts independently.
MTBF / MTTR
Mean Time Between Failures and Mean Time To Recovery — the two numbers that together describe how often a system fails and how fast it comes back.

N

N+1 problem
A performance bug where code issues one query to fetch a list, then one additional query per item to fetch related data, instead of a single joined/batched query.
Normalization
Structuring relational data to minimize duplication by splitting it into related tables — the opposite trade-off from denormalization.

O

Optimistic locking
A concurrency strategy that lets multiple transactions proceed without locking, then checks for conflicts (e.g., via a version number) at commit time.
Outbox pattern
Writing a state change and the event describing it in the same local database transaction, then reliably publishing the event separately — avoids the dual-write problem between a DB and a message broker.

P

Pagination
Returning a large result set in smaller pages instead of all at once — see also cursor-based pagination.
Partitioning / sharding
Splitting a dataset across multiple database nodes by some key, so no single node has to hold or serve all the data. See Databases.
Pessimistic locking
A concurrency strategy that locks a row/resource before modifying it, blocking other transactions until the lock is released — safer under contention, costs throughput.
Polling
A client repeatedly asking a server "is there anything new?" on a fixed interval, as opposed to the server pushing updates.
Publish-subscribe (pub/sub)
A messaging pattern where publishers send messages to a topic without knowing the subscribers, and any number of subscribers can receive them independently. See Observer pattern.

Q

QPS (Queries Per Second)
The number of requests a system handles per second — the core unit of throughput used in capacity estimation.
Quorum
The minimum number of nodes in a distributed system that must agree (or acknowledge) for a read or write to be considered successful, used to balance consistency and availability.

R

Rate limiting
Restricting how many requests a client can make in a given time window, protecting a system from overload or abuse. See HLD Capstones.
Read replica
A copy of a database that serves read traffic, kept in sync with a primary that handles writes, so read load can scale independently of write load.
Retry with jitter
Adding a small random delay to retry backoff timing so that many clients retrying after the same failure don't all hit the server at exactly the same moment (the "thundering herd").
Reverse proxy
A server that sits in front of one or more backend servers and forwards client requests to them, often also handling TLS termination, caching, and load balancing.

S

Sharding
See Partitioning.
Singleton
A design pattern that restricts a class to a single instance with a global access point. See the Singleton pattern page.
SLA / SLO / SLI
Service Level Agreement (the promise, often contractual), Objective (the internal target, e.g., 99.9% uptime), and Indicator (the actual measured metric) — three layers of the same reliability conversation.
Sticky session
A load balancing behavior where all requests from a given client are routed to the same backend server, often used to keep session state in server memory.
Strong consistency
A model where every read reflects the most recent write, everywhere, immediately — simpler to reason about, but costs coordination and latency.

T

Throughput
The total amount of work a system completes per unit of time (e.g., requests/second) — distinct from latency, which is about a single request's speed.
TTL (Time To Live)
How long a piece of cached or DNS data is considered valid before it must be refreshed or re-fetched.
Two-phase commit
A protocol for committing a transaction across multiple nodes atomically: a "prepare" phase where all nodes vote, then a "commit" phase where all apply the change (or all abort).

U

Upsert
A single database operation that inserts a row if it doesn't exist, or updates it if it does — avoids a separate check-then-write round trip.
UUID
A 128-bit identifier that's globally unique with extremely high probability without needing a central coordinator to assign it — common for distributed ID generation.

V

Vertical scaling
Adding more resources (CPU, RAM) to a single machine rather than adding more machines. See Core Principles.
Vector clock
A mechanism for tracking causality between events across distributed nodes without relying on synchronized wall-clock time.

W

Webhook
A server-to-server HTTP callback: instead of polling for an update, a service POSTs to a URL you registered the moment an event happens.
WebSocket
A protocol providing a persistent, full-duplex connection between client and server, used for real-time features like chat. See the WhatsApp case study.
Write-ahead log (WAL)
A durability technique where a change is written to an append-only log before it's applied to the main data structure, so a crash mid-write can be recovered from by replaying the log.

Z

Zero-downtime deployment
A deployment strategy (e.g., rolling or blue-green) where new versions of a service replace old ones without any window of unavailability for users.
