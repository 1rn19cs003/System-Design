Decorator Pattern

Attaches additional responsibilities to an object dynamically, by wrapping it — a flexible alternative to subclassing for extending behavior.

Theory

The problem, in plain terms

You're building a coffee shop ordering system. A Coffee can have milk, whipped cream, caramel, extra shots — any combination, each adding to price and description. Model this with subclasses and you get CoffeeWithMilk, CoffeeWithMilkAndCream, CoffeeWithMilkAndCreamAndCaramel — a class explosion driven by optional add-ons.

Decorator wraps objects instead of subclassing them. Each add-on (Milk, Cream, Caramel) is its own small class wrapping a Coffee (or another decorator), implementing the same Coffee interface, adding its own contribution before or after delegating to what it wraps. You build the final coffee by wrapping layer after layer at runtime.

How it's built

A Component interface (Coffee) is implemented both by concrete base objects (Espresso) and by Decorator classes that also implement Component while holding a reference to a wrapped Component. Because every decorator satisfies the same interface as what it wraps, decorators stack arbitrarily and the caller can't tell how many layers deep the object is.

✓ Good practice

Keep each decorator focused on one responsibility (one topping, one behavior) — that's what makes arbitrary stacking in any combination actually work cleanly.

✕ Common mistake

Not accounting for order-dependence. Wrapping a discount decorator before vs. after a tax decorator gives different final prices — decorator order isn't always cosmetic.

Where it bites you

Deeply nested decorator chains can get hard to read and debug — "what does this object actually do?" requires mentally unwinding several layers. It's also easy to over-decorate something that would be clearer as one configurable object with a few flags.

Diagram

When to Use

✓ Reach for it when
  • You need optional, combinable behavior added at runtime, and subclassing every combination would explode the class count.
  • The added behavior should be transparent — callers shouldn't need to know or care that the object is decorated.
✕ Don't reach for it when
  • Only one or two fixed combinations exist and never change — a couple of subclasses or a flag is simpler.
  • The "decoration" is actually a different interface — that's Adapter, not Decorator.

Real-World Examples

  • Java I/O streams — new BufferedReader(new InputStreamReader(new FileInputStream(...))) layers buffering and decoding onto a raw byte stream.
  • Python decorators (@staticmethod, @property, custom decorators) — wrapping a function to add behavior while preserving its callable interface.
  • UI component libraries — wrapping a base component with Scrollable/Border/Tooltip decorators that add extras while remaining a valid component.
  • HTTP middleware chains (Express.js, servlet filters) — each middleware wraps the next handler, optionally adding behavior before/after calling through.
  • Coffee shop / pizza ordering systems — the canonical teaching example for stacking add-ons without subclassing every combination.

Interview Questions

Click a question to reveal the answer.

How does Decorator differ from just using inheritance to add behavior?

Inheritance fixes behavior combinations at compile time — one subclass per combination, exploding as options grow. Decorator adds behavior at runtime by wrapping objects, so any combination and order is possible without pre-declaring a subclass for it.

How is Decorator different from Adapter, given both wrap another object?

Decorator preserves the same interface as what it wraps — a decorated Coffee is still usable anywhere a Coffee is expected, and any number of decorators can stack. Adapter changes the interface specifically because the wrapped object's original interface doesn't match what the caller needs.

Does the order in which you apply decorators matter?

Yes, often. Wrapping a "10% discount" decorator around a "tax" decorator gives a different final price than wrapping tax around discount. This is a common follow-up question to test whether a candidate has actually implemented the pattern, not just memorized the diagram.

Give a real-world example of Decorator outside "coffee shop" toy examples.

Java's I/O stream classes: new BufferedReader(new InputStreamReader(new FileInputStream(...))). Each wrapper adds one capability (buffering, byte-to-character decoding) while implementing the same stream interface as what it wraps, and layers can be added, omitted, or reordered as needed.
