OOP Fundamentals

The four ideas — encapsulation, abstraction, inheritance, and polymorphism — that every design pattern in this guide is ultimately built out of.

In Plain English

Think about driving a car without knowing anything about engines.

New to this? Start here

You never need to know how the engine works to drive — you just need the pedal and the wheel to behave predictably. That's encapsulation and abstraction: hide the complicated part, expose a small, simple interface.

Already comfortable? Push further

The interesting part is that the same pedal works whether the engine burns gas or draws from a battery — that's polymorphism. And an SUV reusing everything a Car already defines, while adding its own extras, is inheritance.

Theory & Diagrams

The four pillars

Object-oriented programming organizes code around objects — bundles of data and the behavior that acts on that data — rather than around a sequence of standalone functions. Four ideas make that useful in practice.

Encapsulation vs. abstraction

These two get conflated constantly. Encapsulation is a mechanism — bundling data with the methods that touch it, and restricting direct access to that data. Abstraction is a design goal — deciding what to expose and what to hide, so a caller only ever has to think about the "what," never the "how."

Inheritance and the fragile base class problem

A subclass reuses a parent's fields and methods and can override or extend them. That reuse is powerful, but it also means a change to the parent can silently break every subclass depending on its exact behavior — the deeper and wider the hierarchy, the more fragile it gets.

Polymorphism

Calling shape.area() on a variable typed as the base class Shape runs Circle's implementation, Rectangle's implementation, or whichever subclass the object actually is — decided at runtime, not at compile time. This is what lets code work with a list of mixed shapes without a single if/else chain checking types.

Trade-offs

✓ Favor inheritance when
  • There's a genuine, stable "is-a" relationship (a Square really is a Rectangle with equal sides).
  • Subclasses are expected to share and rarely override the parent's core behavior.
✕ Favor composition when
  • The relationship is really "has-a" or "can-do," not "is-a" (a Car has-an Engine, it isn't one).
  • You need to swap behavior at runtime, or the hierarchy would otherwise need multiple inheritance.

What interviewers are listening for: a precise distinction between encapsulation and abstraction (not "they're basically the same thing"), and knowing the "favor composition over inheritance" principle by name along with a concrete example of when a hierarchy became a liability.

If you're a fresher: being able to define all four pillars in one sentence each, with a simple example per pillar, covers what most early interviews check for.

If you're ~3 years in: be ready to describe a real inheritance hierarchy you've worked with that became painful to change, and what composition-based fix you'd reach for instead.

Real-World Examples

  • GUI frameworks (Swing, WPF, UIKit) — every widget inherits from a common base (like View or Component), and rendering code calls the same draw() method polymorphically across wildly different widget types.
  • ORMs (Hibernate, SQLAlchemy, Entity Framework) — model classes encapsulate both the data and the logic to load/save it, abstracting the caller away from raw SQL.
  • Game engines — a generic GameObject base class with subclasses like Player, Enemy, and Projectile, all updated through the same polymorphic update() call each frame.
  • Standard library collections (Java's List, Python's iterables) — abstraction lets code iterate any collection type through one shared interface, without caring whether it's an array, linked list, or set underneath.

Interview Questions

Click a question to reveal the answer.

What's the difference between encapsulation and abstraction?

Encapsulation is the mechanism: bundling data with the methods that operate on it, and restricting direct access (private fields, public methods). Abstraction is the design decision of what to expose versus hide, so callers only need to know "what" an object does, not "how." Encapsulation is how you achieve abstraction, but the two aren't the same thing.

What does "favor composition over inheritance" mean, and why?

It means preferring to build behavior by holding references to other objects (composition) rather than by extending a base class (inheritance), when possible. Composition avoids the fragile base class problem, allows swapping behavior at runtime, and avoids forcing an "is-a" relationship onto something that's really "has-a."

What's the difference between compile-time and runtime polymorphism?

Compile-time (static) polymorphism is method overloading — the compiler picks which method to call based on argument types at compile time. Runtime (dynamic) polymorphism is method overriding — the actual method that runs is decided by the real object type at runtime, which is what enables working with a list of mixed subclasses through one shared interface.

What's the fragile base class problem?

When subclasses depend on the exact internal behavior of a parent class, a seemingly safe change to that parent can silently break every subclass relying on it — even though nothing about the subclass's own code changed. Deep or wide inheritance hierarchies make this worse the more classes depend on the same shared parent.
